# KingdeeKB 项目坑点、解决方案与技术瓶颈复盘

> 日期：2026-06-08  
> 范围：KingdeeKB 桌面端、知识库、Agent、技能系统、风控、调研、产物生成、Tauri/Rust/React 工程化  
> 用途：项目复盘、技术汇报、后续治理计划输入

---

## 1. 总体结论

KingdeeKB 的主要难点不在单点功能，而在“AI 副驾 + 本地知识库 + 业务工作台 + 可治理工具执行”同时落地。项目过程中遇到的坑大致分为四类：

1. **AI 行为不可控**：只靠 prompt 约束不够，工具、模型、项目、技能都必须在后端强制校验。
2. **状态不可恢复**：流式对话、工具调用、澄清、取消和错误如果只存在前端内存，刷新或失败后无法追责。
3. **数据边界复杂**：知识库、附件、项目、调研、风险、产物都要隔离，不能靠文件名或文本规则猜测。
4. **工程面复杂度高**：Tauri 命令、SQLite、Rust async、React 状态、Playwright mock、Windows 环境都会放大细节问题。

最终解决方向是：把“能跑”升级为“有事实源、有治理策略、有审计、有恢复能力”。

---

## 2. 关键坑点与解决方式

### 2.1 页面归属误判

坑点：

- 看到目录名或页面文案后，容易误判目标页面。
- 曾出现把功能改到相似页面，而不是路由真正挂载组件的情况。

影响：

- 改动看似完成，但用户实际访问页面没有变化。
- 增加回归成本，也容易引入无关 UI 变更。

解决：

- 明确规则：新增或修改页面前，必须先查路由表，例如 `Layout.tsx`、`App.tsx` 或路由配置。
- 从路由找到真实组件后，再读完整组件代码。
- 禁止只凭文件名、目录名、中文文案判断页面归属。

沉淀：

- 已写入 `AGENTS.md` 的“路由验证原则”。

### 2.2 旧协议兼容导致语义混乱

坑点：

- 项目尚未发布，但早期实现容易保留旧模板、旧协议、双格式读取或自动猜测。
- 这会让后续数据结构越来越难收敛。

影响：

- 后端逻辑出现多套语义。
- 测试需要覆盖旧协议，增加维护成本。
- AI 工具调用时可能引用已经废弃的模板或协议。

解决：

- 确认项目未发布后，采用“直接替换当前实现”的原则。
- 删除旧模板向导语义，转向技能产物协议。
- 配置、测试、本地开发数据直接更新为当前格式。

沉淀：

- `AGENTS.md` 中明确禁止旧实现兼容、迁移猜测和双协议尝试。

### 2.3 Agent 只靠 prompt 控制不可靠

坑点：

- 初期容易把项目隔离、工具禁用、技能执行规则写进系统提示词。
- 但模型可能忽略、误解或在多轮上下文中绕过这些规则。

影响：

- 禁用工具仍可能被模型尝试调用。
- 未授权脚本执行、跨项目搜索、错误 provider 选择都有风险。

解决：

- 将约束下沉到工具层和服务层。
- `rig_tool.rs` 增加工具 profile、schema guard、输出截断、审计记录、禁用策略。
- Provider Policy 在后端运行态强制过滤 provider/model，不仅仅依赖前端隐藏。

沉淀：

- prompt 只表达意图和解释策略，强制约束必须在后端执行。

### 2.4 Agent 会话不可恢复

坑点：

- 对话流、工具调用、澄清问题、取消状态主要存在前端内存或 localStorage。
- 刷新、取消、异常中断后无法完整还原“发生了什么”。

影响：

- 无法审计 AI 做过哪些工具调用。
- 无法解释某个回答来自哪些工具结果。
- 失败后用户难以继续上一次任务。

解决：

- 新增 Agent Session Ledger：
  - `agent_sessions`
  - `agent_messages`
  - `agent_tool_calls`
  - `agent_tool_results`
  - `agent_events`
- `agent_chat` 启动、用户消息、assistant 流式输出、工具调用、工具结果、澄清、取消、错误、完成都写入账本。
- Chat 页面刷新后优先从后端账本恢复当前项目最近一次 `chat` 会话，失败时再回退 localStorage。

沉淀：

- Ledger 成为 Agent 恢复、审计、验证的事实源。

### 2.5 工具审计缺少会话身份

坑点：

- 工具审计最初能记录工具名、耗时、状态、输出截断，但不能稳定关联到用户消息、assistant 消息和会话。

影响：

- 只能看“某个工具执行过”，不能回答“谁触发的、在哪次对话中触发的、属于哪个工具调用”。

解决：

- 工具审计 JSONL 增加：
  - `session_id`
  - `assistant_message_id`
  - `tool_call_id`
- 运行时工具守卫为每次工具执行生成独立 `tool_call_id`。
- 账本中写入 `tool_revision=rig-tool-profile-v1`，后续用于工具 profile 版本校验。

沉淀：

- 审计记录必须能从工具结果追溯到 session/message/tool_call。

### 2.6 模型供应商配置存在但不代表允许使用

坑点：

- 早期 provider 配置和运行授权混在一起。
- 只要配置存在，Agent、图片处理、多模态候选、默认模型选择就可能使用它。

影响：

- 管理员禁用某个 provider 后，后端仍可能通过默认模型或多模态分支绕过。

解决：

- 增加 Provider Policy：
  - `allow`
  - `deny`
  - 支持 provider 级、model 级资源规则。
- 后端在以下路径强制检查：
  - 默认运行 provider
  - 指定 provider/model
  - 多模态候选
  - 模型列表
  - 连接配置
- 前端 Chat 模型选择只展示运行态允许的 provider/model。

沉淀：

- 配置存在不等于运行可用，运行态必须过 policy。

### 2.7 LLM 输出上限导致工具参数被截断

坑点：

- rig 的 `max_tokens()` 控制生成输出，不是上下文容量。
- 技能工作流可能把 HTML、PPT 输入等长内容放进工具参数，默认输出预算容易截断 tool call。

影响：

- 模型生成的工具参数 JSON 不完整。
- 工具调用失败，用户看到的是“工具没跑起来”，实际是模型输出被截断。

解决：

- Agent 输出 token 预算设置工程下限和上限：
  - 最低 16K
  - 最高 32K
- 对输出上限错误做专门识别，返回可理解的错误提示。

沉淀：

- UI 的“上下文容量”和模型“生成输出上限”要区分。

### 2.8 工具输出过大撑爆上下文

坑点：

- 知识检索、脚本执行、文件生成工具可能返回大量文本。
- 如果完整塞回模型上下文，会导致成本高、速度慢、甚至上下文溢出。

影响：

- Agent 响应变慢。
- 后续推理基于超长工具输出，稳定性下降。

解决：

- 工具输出增加统一截断规则：
  - 最大字符数
  - 最大字节数
  - 最大行数
- 完整输出落到 `agent_tool_outputs`，返回给模型的是预览和保存路径。
- 设置页提供 Agent 工具审计和输出读取能力。

沉淀：

- 工具结果必须有“模型可消费预览”和“人工可追溯完整输出”两层。

### 2.9 工具调用死循环

坑点：

- ReAct 模式下模型可能连续调用相同工具和相同参数。

影响：

- 浪费 token 和时间。
- 用户看不到有效进展。

解决：

- 增加最近工具调用队列。
- 如果连续多次工具名和参数完全一致，判定死循环并中断。
- 增加每分钟工具调用上限。

沉淀：

- Agent 必须有预算、速率和重复调用保护，不能无限信任模型自我停止。

### 2.10 澄清问题取消流程容易悬挂

坑点：

- `question` 工具会阻塞等待用户回答。
- 用户取消澄清后，如果后端 pending question 没有正确释放，会导致会话卡住。

影响：

- 前端看似取消，后端仍等待。
- 再次回答或继续对话时状态不一致。

解决：

- 增加 `reject_question` 后端命令。
- 前端取消澄清时调用后端拒绝接口，释放 oneshot。
- 取消和拒绝也进入 Agent ledger。

沉淀：

- 用户交互型工具必须同时设计“回答”和“拒绝/取消”路径。

### 2.11 本地聊天历史不是可靠事实源

坑点：

- localStorage 只能保存 UI 消息，无法保存工具调用、事件、错误、审计身份。

影响：

- 前端刷新能看到文字，但无法恢复执行上下文。
- 审计和恢复能力不足。

解决：

- localStorage 降级为 fallback。
- 以后端 ledger 作为恢复入口。

沉淀：

- UI 缓存不能替代后端事实源。

### 2.12 知识来源引用依赖文本正则

坑点：

- 前端从 `search-knowledge` 工具结果的中文文本里用正则提取来源。

影响：

- 一旦工具输出格式变化，sources 解析就失败。
- 来源链路不稳定，不利于审计。

解决：

- 已在成熟度计划中明确下一步改为结构化 sources。
- 工具结果应返回 `source_type`、`source_id`、`project_id`、`title`、`snippet`、`score`、`trusted` 等字段。

沉淀：

- Grounding 来源不能依赖自然语言文本解析。

### 2.13 项目隔离不能只靠搜索参数

坑点：

- 文档、chunks、Wiki、附件、调研、产物、风险都和项目相关。
- 如果某条路径漏传 `project_id`，就可能跨项目读取。

影响：

- 企业场景下属于严重数据边界问题。

解决：

- 数据库表逐步统一到 `project_id`。
- 搜索、附件入库、风险分析、产物登记都绑定当前项目。
- Agent 系统提示词中说明当前项目，但真正隔离仍在工具参数和数据库查询层执行。

沉淀：

- 项目隔离必须贯穿 schema、服务层、工具层和前端状态。

### 2.14 文档摄入并发与向量 key 分配

坑点：

- 使用 `MAX(vector_key)+1` 这类方式分配向量 key，在并发摄入时存在冲突风险。

影响：

- chunks 和向量索引可能错位。
- 重建索引和删除文档时难以保证一致性。

解决：

- 使用 `vector_key_seq` 表，通过 SQLite 自增行分配全局唯一 vector key。
- 元数据层增加 `next_vector_key()`。

沉淀：

- 并发唯一 ID 不能靠查询当前最大值推导。

### 2.15 文档去重只看 sha256 会混淆来源

坑点：

- 两份内容相同但来源不同的文件，如果只按 sha256 定位，可能返回错误 document_id。

影响：

- 删除、更新、raw_source 级联清理时可能误伤。

解决：

- 文档定位改为 `sha256 + project_id + raw_source_identity`。
- 相关查询方法按来源键精确查找。

沉淀：

- 内容相同不等于业务来源相同。

### 2.16 技能脚本执行安全边界复杂

坑点：

- 技能脚本可能读取文件、联网、生成产物、安装依赖。
- 如果直接让模型拼 shell 命令，风险不可控。

影响：

- 脚本越权读取。
- 依赖安装污染系统环境。
- 产物路径和输入来源不可追踪。

解决：

- 要求必须先 `use-skill` 再 `run-skill-script`。
- 脚本只能来自 skill 的 `scripts/` 目录。
- 参数禁止 shell 控制符。
- 每次执行在独立沙箱目录中，产物从沙箱登记。
- 下一步继续补 `SkillNetwork`、`SkillExternalRead` 权限规则。

沉淀：

- 技能脚本不是普通工具，而是受控 action。

### 2.17 Windows PowerShell 命令容易被管道拆分影响

坑点：

- 在受限沙箱下，包含管道、重定向、复杂 shell 语法的命令可能被拆分为多个 segment。
- 一些读取命令因为 PowerShell 管道触发执行失败。

影响：

- 调试时容易误以为文件或命令有问题。

解决：

- 优先用 `rg` 定位。
- 避免把多个读取命令串在一个 shell 里。
- 并行读取使用 `multi_tool_use.parallel`，每个 shell 命令保持简单。

沉淀：

- Windows 环境下，命令要短、明确、少组合。

### 2.18 前端流式更新容易造成重渲染压力

坑点：

- 每个 token 都 setState 会导致 React 高频渲染。

影响：

- 长回答时 UI 卡顿。
- 消息列表滚动不稳定。

解决：

- 对 `text_delta` 和 `thinking` 做 requestAnimationFrame 缓冲。
- 非流式事件到来前先 flush buffer。

沉淀：

- 流式 UI 要做批量刷新，不能逐 token 刷状态。

### 2.19 E2E mock 漏命令会导致页面测试失败

坑点：

- 新增 Tauri 命令后，浏览器 E2E 环境没有真实后端。
- 如果 mock 没补，页面一加载就失败。

影响：

- 功能实现正确但测试挂在 mock 缺口。

解决：

- 新增命令同步更新 `e2e/mocks/tauri-mock.ts`。
- Provider Policy、Agent Ledger 命令都补默认 mock。

沉淀：

- Tauri command 新增必须同步 mock 和类型封装。

### 2.20 脏工作树下提交风险高

坑点：

- 项目中长期存在大量未提交改动和删除。
- 有些文件同时包含本轮改动和历史改动。

影响：

- 文件级 `git add` 容易把无关改动提交进去。
- 自动格式化可能扩大 diff。

解决：

- 修改前先看 `git status`。
- 提交前必须按文件和 hunk 核对。
- 对重叠文件保持谨慎，不自动回滚用户改动。

沉淀：

- 脏工作树里默认不做“顺手提交”，除非能明确隔离改动。

## 3. 技术瓶颈与突破方式

### 3.1 AI 副驾成熟度瓶颈

瓶颈：

- 项目已经能对话、检索、调用工具，但距离成熟企业副驾还差治理、恢复、审计。

突破：

- 调研 Microsoft 365 Copilot、Salesforce Agentforce、ServiceNow AI Control Tower、SAP Joule、Gemini、Glean、Rovo、ChatGPT Enterprise 等成熟产品。
- 提炼统一成熟度模型：
  - 权限继承
  - grounding 可解释
  - 动作可治理
  - 会话可恢复
  - 审计可追责
  - 质量可评估
  - 业务闭环

结果：

- 形成 `2026-06-08-ai-copilot-maturity-research-and-supplement-plan.md`。
- 当前已开始按 P0 治理底座推进。

### 3.2 ReAct Agent 稳定性瓶颈

瓶颈：

- 模型工具调用存在循环、参数错误、输出过长、用户交互阻塞、取消难恢复等问题。

突破：

- 使用 rig 原生 function calling。
- 增加工具 schema guard。
- 增加工具输出截断。
- 增加调用速率限制和重复调用检测。
- 增加 clarification/reject/cancel 流程。
- 增加 session ledger。

结果：

- Agent 从“能调用工具”逐步转向“可治理工具执行”。

### 3.3 本地知识库一致性瓶颈

瓶颈：

- 文档摄入、向量索引、BM25、SQLite 元数据必须保持一致。
- 并发摄入、重复文件、删除重建都会影响一致性。

突破：

- 元数据统一进入 SQLite。
- 向量 key 使用自增序列表。
- 文档来源使用 `raw_source_identity` 精确绑定。
- 删除通过 outbox 和索引重建逐步补齐一致性。

结果：

- 文档、chunks、向量 key 和项目边界更可控。

### 3.4 多模型与多供应商治理瓶颈

瓶颈：

- OpenAI 兼容、Anthropic、本地 Ollama、多模态候选、OCR 回退等路径分散。
- 仅靠默认 provider 容易绕过管理员策略。

突破：

- 引入 Provider Policy。
- 运行态 provider 列表与配置列表分离。
- 后端所有模型选择路径统一检查 allow/deny。

结果：

- 禁用 provider 后，前后端都不能继续使用该 provider。

### 3.5 技能产物生成瓶颈

瓶颈：

- 企业实施交付物不是纯文本回答，需要真实文件输出。
- 脚本执行涉及依赖、输入文件、输出目录、产物登记、安全授权。

突破：

- 统一使用 skill 目录和 `scripts/`。
- run-skill-script 在沙箱中执行。
- 输出文件按扩展名收集并登记产物。
- 工具返回摘要，完整输出进入审计。

结果：

- 从“聊天生成内容”转向“生成可管理交付物”。

### 3.6 前后端类型与命令一致性瓶颈

瓶颈：

- Tauri 命令多，前端 TS 类型、Rust 参数、mock 三处容易不一致。

突破：

- 新增命令时同步：
  - Rust command
  - `lib.rs` invoke handler
  - TS command wrapper
  - TS 类型
  - E2E mock

结果：

- `cargo check`、`npx tsc --noEmit`、Biome、Playwright 形成基本质量门禁。

## 4. 后续仍需解决的问题

1. **结构化 sources**  
   当前部分来源还依赖文本解析，后续要统一返回结构化来源。

2. **技能脚本权限细化**  
   需要补 `SkillNetwork`、`SkillExternalRead`、解释器、路径、联网和输入输出权限展示。

3. **风控 evidence 结构化**  
   风险结论需要带证据链，并能登记为产物或审计记录。

4. **工具 profile 强版本校验**  
   当前账本记录了 `tool_revision=rig-tool-profile-v1`，后续要推进到 runtime guard 的强校验。

5. **Agent 诊断页**  
   需要集中展示 provider、模型能力、工具启停、权限规则、最近失败、预算使用。

6. **回归集和 UAT**  
   需要把知识问答、技能执行、调研澄清、风险判断、取消恢复做成固定回归集。

## 5. 可复用经验

1. AI 项目不要只问“模型能不能答”，要问“答错、越权、失败、中断后能不能解释和恢复”。
2. 所有高风险能力必须后端强制校验，prompt 只能作为辅助说明。
3. 会话、消息、工具、结果、事件必须有统一事实源。
4. 工具输出必须区分模型可消费预览和人工可追溯完整内容。
5. 页面改动必须先查路由，避免改错组件。
6. 新增 Tauri 命令必须同步 Rust 注册、TS 类型、前端封装和 E2E mock。
7. 项目未发布时，不要为旧协议背包袱，直接收敛到单一当前语义。
8. 脏工作树下提交必须按 hunk 核对，不能粗暴文件级提交。
