# 交付物扫描统一规范

> 本文件是套件内交付物盘点的**唯一口径来源**。
> project-dashboard（Python 实现 `server/deliverable_scanner.py`）与
> acceptance-pack（Skill 流程）均以本规范为准，确保对同一项目扫出的清单一致。

---

## 1. 阶段目录映射

7 个业务阶段目录 → 内部简称：

| 目录 | 简称 |
|------|------|
| `01_启动阶段` | 启动 |
| `02_需求阶段` | 需求 |
| `03_方案阶段` | 方案 |
| `04_构建阶段` | 构建 |
| `05_测试阶段` | 测试 |
| `06_上线阶段` | 上线 |
| `07_验收阶段` | 验收 |

`00_项目管理` 为跨阶段管理目录，盘点时单列「项目管理」分组。
目录名前缀可能有变体（`01_` vs `1_`），匹配用包含判断，不依赖精确名。

---

## 2. 文件排除规则

扫描阶段目录时**排除**以下文件，不计入交付物：

- 隐藏文件：`.` 开头（`.DS_Store`、`.dashboard-signals.jsonl` 等）
- 系统/临时文件：`Thumbs.db`、`~$` 开头（Office 锁定文件）
- 模板文件：文件名含「模板」或「_模板」
- `被裁剪文件/` 目录下的全部内容
- 系统目录：`.git/`、`.claude/`、`node_modules/`

递归扫描深度建议 ≤ 3 层子目录。

---

## 3. 必选清单查找优先级

按顺序查找必选交付物清单，命中即用：

1. `00_项目管理/简化版必选交付物清单.xlsx`（openpyxl 解析；首列=阶段，次列=交付物名称，可选「是否必选」列）
2. `00_项目管理/必选交付物清单.md`（markdown 表格）
3. fallback：`project-dashboard/references/default-deliverables.md`（7 阶段标准必选清单）

---

## 4. 交付物状态判定（4 态）

对每个交付物单元，状态取值：

| 状态 | 判定条件 |
|------|---------|
| `done` | 文件存在 **且** 经客户签字（signals.jsonl 有对应 `deliverable.completed` 且 `signed_off=true`） |
| `pending` | 文件存在但未签字确认 |
| `missing` | 必选清单要求、但阶段目录中找不到匹配文件 |
| `optional` | 非必选，且清单未覆盖的已有文件 |

**文件匹配**：交付物名称与文件名做模糊匹配——文件名（去扩展名）包含交付物名称，或交付物名称包含文件名，任一成立即视为命中。

**无必选清单时**：所有已扫到的文件按 `pending` 处理，不产生 `missing`（不知道该有什么，就不报缺失）。

---

## 5. 一致性要求

- project-dashboard 的交付物网格、acceptance-pack 的「项目文档清单」与「缺失交付物」分析，必须使用本规范同一套排除规则与状态判定。
- 任一方调整扫描口径时，先改本文件，再改各自实现。
